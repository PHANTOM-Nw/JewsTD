# 敌人队列与体型推动记录

日期：2026-07-14

## 目标

让减速效果产生可读的队伍拥堵：减速前兵会挡住后兵，但可被体型大于等于自己的后兵有限推动，避免所有敌人完全重叠穿行，也避免减速队列永久停滞。

## 实现

- 新增 `engine/enemyMovement.ts` 纯计算边界，把离散路径游标转换为连续路程，并在还原游标时保留跨路径段的剩余位移。
- 每帧先统一更新毒素、减速和眩晕计时并收集自由位移，再按路径从前到后解析敌人间距，最后一次性回写位置。
- 敌人最小间距读取 `config/enemies.ts` 中双方体型半径与 `ENEMY_TRAFFIC_CONFIG.gap`；运行时不维护第二份体型数值。
- 所有追尾敌人都会在前兵身后排队。只有仍受减速影响且未眩晕的前兵可以被推动，且后兵体型必须大于等于前兵；推动按双方体型加权，并限制在前兵本帧基础速度上限内。眩晕保持不可推动的硬控制。
- 每名成功生成的敌人获得局内单调递增的 `spawnSequence`。路程相同时用该序号稳定决定前后，不依赖敌人数组的当前顺序。
- 入口净空检查与队首状态转换保持纯函数：空间不足时不移除到期队首、不递增序号，后续帧继续重试；一次成功生成后，新敌人立即参与净空检查，防止同帧补出重叠敌人。
- 死亡和已漏怪敌人不会参与移动队列或入口占位。到达终点的敌人在同一帧统一结算矿坑伤害、标记并移出活动数组，避免重复扣除。

## 验证

- Vitest 当前共 21 个测试文件、146 项测试。
- 新增回归覆盖路径跨段余量、较小后兵阻挡、同体型和较大后兵推动、连续多帧队列前进、体型推动链、眩晕硬阻挡、正常单位不可被推动、稳定生成序号、入口延迟且不丢队首或重复计数、同帧漏怪批量结算与终点释放。
- `npm run lint`、`npm test`、`npm run build` 和 `git diff --check` 作为交付验证执行。
- 390×844 headless Chromium 启动烟测确认应用、地图路线与建造界面正常加载。当前没有可注入指定敌人、减速和队列状态的浏览器调试入口，实际减速拥堵还依赖随机获得减速塔并进入密集波次，因此本轮未把该场景表述为浏览器实战已验证；确定性队列规则由纯函数回归保护，仍需后续完整波次实机观察。
