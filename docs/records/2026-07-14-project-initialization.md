# 项目初始化与结构整理

## 背景

仓库由 Qoder 初步生成，已有可玩的宝石 TD 原型，但根目录堆积了多份报告，README 仍是 Vite 模板，游戏代码散落在通用目录中，且没有统一的目录和文档约定。

## 发现

- 技术栈为 React 19、TypeScript、Vite 和 Oxlint。
- 核心引擎集中在单个大型 Hook 中，UI state 与高频 ref 状态的分层思路合理，但模块职责过多。
- 两个可视化检查组件与正式组件混放；所谓 `pathfinding.test.ts` 是手工函数集合，没有测试运行器。
- 初始 TypeScript 构建存在 15 个错误，lint 有 9 条警告。
- 塔随机品质使用 `regular/polished`，但运行时塔配置只接受 `GemLevel` 的 `flawed/normal`，会在随机到高品质时读取不到配置。
- 主组件直接调用不存在的 `setUiState` 删除障碍物，动作边界不完整。
- 多份旧文档只描述 4 种宝石，与当前 8 种宝石实现不一致。

## 决定与变更

- 源码整理为 `app/`、`game/` 和 `devtools/`，游戏域内再按组件、配置、引擎、寻路、服务、类型和工具拆分。
- 把删除障碍物收敛为引擎动作，组件只调用公开动作。
- 统一随机品质和领域品质类型，当前随机生成碎裂、有瑕、普通三档。
- 长期文档按产品、架构、参考和开发约定分类；Qoder 生成报告集中归档。
- 不在本次直接拆分大型引擎 Hook，先记录技术债，避免无测试保护下的大规模行为改写。

## 验证

- `npm run build`：通过。
- `npm run lint`：通过，无警告。

## 后续事项

1. 引入正式测试框架，把寻路手工检查转成自动化测试。
2. 逐步将战斗、波次和合成逻辑从 `useGameEngine` 提取为纯函数。
3. 建立游戏内通知和调试日志开关，替换 `alert` 与大量控制台输出。
4. 为 Canvas 和控制面板补响应式布局与视觉规范。
