# 游戏配置入口

运行时配置以 `src/game/config/` 为唯一事实来源。修改数值时直接编辑对应 TypeScript 文件，并让类型系统帮助发现遗漏；不要维护第二份完整数值表。

| 文件 | 职责 | 典型修改 |
| --- | --- | --- |
| `map.ts` | 网格尺寸、起点、终点、必经点 | 地图布局和路径形状 |
| `economy.ts` | 起始木材/金币/生命、每轮木材、建造/清障成本、障碍上限 | 回合资源与经济节奏 |
| `towers.ts` | 基础塔属性、特殊塔配方、显示名、颜色、随机品质概率 | 塔平衡和新塔 |
| `enemies.ts` | 敌人生命、速度、护甲、魔抗、奖励 | 敌人平衡 |
| `waves.ts` | 每波敌人组合、生成间隔、生命倍率 | 难度曲线 |

领域类型位于 `src/game/types/game.ts`。添加新宝石、特殊塔或敌人类型时，先扩展类型，再补齐配置和所有 `Record<...>` 映射。当前配置包含 8 种基础宝石、6 种特殊塔、4 种敌人和连续 12 波；第 12 波必须保留 `isBossWave` 标记并包含 Boss 生成组。

当前地图是 10 行×8 列、每格 40 逻辑像素的竖向棋盘。`map.ts` 中的 `WAYPOINTS` 是路线锚点的唯一坐标来源，`MAP_CONFIG.startPos` 与 `endPos` 从其首尾派生；空图路线为 18 步。`economy.ts` 配置每轮 3 木材、建造 3 座塔、保留 1 座、每次清障 20 金币，并将累计障碍限制为 24 个。建造容量和最老障碍风化规则由 `engine/building.ts` 的纯函数实现，容量检查直接读取当前批次剩余数量和下一轮建造数量，不另设硬编码副本。

当前手机平衡基线为：基础塔射程 100–150，特殊塔射程不超过 145，溅射半径 40–60，邻近穿透搜索半径 60。四类敌人的移动速度与 12 波敌人组合、生命倍率分别以 `enemies.ts` 和 `waves.ts` 为准；难度曲线由第 1 波的 1.0 倍生命递增到第 12 波的 7.0 倍生命，最终波包含唯一 Boss。这里不复制完整数值表，以免与运行时配置形成双重事实来源。

塔配置中的 `multiTarget`、`splashRadius`、`slowEffect`、`critChance`/`critMultiplier`、`pierce`、`poisonDamage`/`poisonDuration` 和 `stunChance`/`stunDuration` 会被引擎带入实战。这里的 `pierce` 当前表示命中后对主目标附近的最近另一个敌人追加伤害；若要改为弹道连续贯穿，需要同时修改战斗实现和测试，不能只改配置注释。

## 修改检查表

1. 确认类型、运行时配置和 UI 显示名同步。
2. 若修改地图尺寸或必经点，确认所有点都在网格边界内、互不重复、首尾与起终点一致，并更新空图路径长度测试。
3. 若修改塔品质概率，确认概率之和为 1，返回值属于 `GemLevel`。
4. 若修改特殊塔配方，确认材料类型和生成属性完整。
5. 若修改经济配置，确认一轮木材仍能支持约定的建造数量，障碍上限与批次安全位规则仍能走完 12 轮，并同步所有资源提示。
6. 若修改敌人或波次，确认类型存在、数量和间隔为正、生命倍率有效，并维护最终 Boss 波约束。
7. 更新对应 Vitest 测试，执行 `npm run lint`、`npm test` 和 `npm run build`。
8. 涉及玩法或模块边界时，同步当前文档并在 `docs/records/` 添加记录。
